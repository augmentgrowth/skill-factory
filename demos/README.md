# Demos

Three walkthroughs you can open in a browser. All are self-contained HTML — no build step,
no server, no dependencies. Open the file and present it.

| Demo | Question it answers | Open |
|---|---|---|
| **1 — What a good skill actually does** | *Why bother?* One transcript into three formats, with and without the skill. | `01-what-a-good-skill-does.html` |
| **2 — Watch a skill get built** | *What happens if I try it?* One full session, intake to committed skill, all seven steps. | `02-build-a-skill-walkthrough.html` |
| **3 — The factory’s first stress test** | *Did the process hold up?* A script-backed two-model workflow skill, built through the factory and judged on a real coding task. | `03-fable-codex-factory-run.html` |

Show them in that order. Demo 1 makes someone want a skill; demo 2 shows them it takes
one conversation to get one; demo 3 lets a skeptical reader audit whether the factory’s
discipline changed a real outcome.

## These are real runs, not mockups

Every output in the gallery was generated by actually running the factory — the no-skill
baselines were captured **before** the corresponding skill existed, so none of the comparisons
is graded on a curve.

That matters most in demo 2, where the baseline is genuinely good: Claude found the story
in the data on its own, flagged a likely tracking bug, and volunteered an attribution
caveat nobody asked for. Nothing was written to lose. The skill still wins, and *why* it
wins is the actual lesson — it carries the target, the intent, and the format that Claude
had no way to know.

## Auditing demo 2

`artifacts/weekly-metrics-update/` holds the run's real output so the claims are checkable
rather than takeable on faith:

```
SKILL.md                          the skill that got built (175 lines)
CHANGELOG.md                      line one, written at birth
cases/baseline/input.md           the frozen sample export
cases/baseline/output-baseline.md the no-skill output — captured before drafting
cases/baseline/output-with-skill.md  the run that closed the done gate
git-history.txt                   both commits and the rollback tag
```

These artifacts are deliberately **not** in `.claude/skills/`. Everything there loads into
every builder's context, and the factory keeps that budget tight — four skills, plus
whatever you build. A demo shouldn't cost you context forever, so it lives here instead,
where nothing auto-loads it.

## Auditing demo 3

`artifacts/fable-codex-factory-run/` holds the frozen task and the full with-skill audit trail.
The built skill graduated to its owner's private build home; `skill/` in this folder is a frozen
snapshot of the SKILL.md, changelog, and cases as they stood at the stress test.

```
task.md                 the frozen pm-agent task
plan.md                 the plan after Sol's accepted revision
loop-transcript.md      every loop stage, verdict, thread state, and final diff stat
plan-review-r1.txt      the exact round-one plan-review prompt
code-review-r1.txt      the exact round-one code-review prompt
final-verdict.txt       the exact fresh-thread final-review prompt
withskill.diff          the complete accepted implementation diff
git-history.txt         birth commit, finished-skill commit, and known-good tag
```

The matching no-skill arm is frozen at
`artifacts/fable-codex-factory-run/skill/cases/baseline/`: `input.md` fixes the task and commit;
`output-baseline.md` preserves the full implementation diff, test result, and run accounting.

## Reusing these

The first two demos are generic on purpose — a weekly metrics update and a podcast
repurpose. Demo 3 is deliberately concrete because its job is auditability, not reuse.
Fork either pattern for your own team: run a real build, capture the before and after,
and let the comparison make the argument.
