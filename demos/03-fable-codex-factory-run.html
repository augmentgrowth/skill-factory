What the factory was asked to build

Not a prompt wrapper. A workflow with scripts, state, role separation, and refusal gates.

fable-codex is a two-model coding loop. Claude orchestrates and arbitrates. GPT-5.6 Sol adversarially reviews the plan and the code in persistent, read-only Codex threads. GPT-5.6 Luna implements in a separate workspace-write thread. No implementation begins until the plan is approved; no run closes until a fresh reviewer thread approves the code.

Orchestrator

Claude

Investigates, writes or ingests the plan, verifies review claims, reads the full diff, fixes directly, and owns the final answer.

Adversarial reviewer

GPT-5.6 Sol

Reviews plans and code in persistent threads, then gives the last gate to a fresh thread with no debate history.

Implementer

GPT-5.6 Luna

Executes the approved plan. It does not review, approve, commit, push, or silently broaden scope.

The test subject was the factory’s own orchestration skill. That made this more demanding than the first two gallery runs: the artifact had executable shell scripts, persistent thread state, sandbox boundaries, verdict parsing, fallback behavior, and exact role prompts.

How the factory actually built it

The order matters. In particular, the baseline existed before the draft.

0

Preflight

The factory checked repository state and authoring prerequisites. One environmental assumption surfaced here: shellcheck was absent and had to be installed during the run.

1

Freeze a real guinea-pig task

A real pm-agent change became the fixture: introduce a validated proposal-kind registry, reject unknown kinds, preserve existing behavior, and leave the suite green. Both arms started from commit 6eec1b8 in disposable worktrees.

2

Capture the no-skill baseline before drafting

A single Fable agent implemented the frozen task with no fable-codex skill and no Codex CLI. It finished with 545 tests green, spent about 117k subagent tokens, and its complete diff was frozen as the bar to beat.

5256b4ffable-codex: birth commit — frozen baseline task + no-skill output
3

Classify it correctly

The factory classified fable-codex as a workflow skill: it orchestrates a multi-stage process and carries internal scripts and references, rather than teaching generic coding knowledge.

4

Draft to the quality bar

Claude authored the orchestration charter and reference prompts. Codex GPT-5.6 Sol implemented the scripts from an exact specification. The script harness passed 14 of 14 stub scenarios and shellcheck; the finished SKILL.md is 188 lines.

5

Self-critique before the gate

Five checks covered voice, principles, anti-patterns, examples, and focus. The weakest point was explicit: no worked end-to-end example in SKILL.md. The run accepted that tradeoff because the reference files carry the exact templates. A malformed code fence was also caught and fixed before testing.

6

Run the literal side-by-side gate

The new skill ran against the same task at the same frozen commit in the second disposable worktree. The question was not “is the skill plausible?” It was “did it beat the already-competent baseline?”

7

Close only after the win

The with-skill arm finished green and received a fresh-thread approval. Only then did the factory record the finished artifact and a per-skill rollback point.

Same frozen task. Same frozen repository commit. Two disposable worktrees. Both implementations passed 545 tests.

The implementation was competent. It built a central registry, rejected unknown kinds before writes, rewired production call sites to constants, added the requested tests, and left the suite green.

The orchestrator checked the helper, confirmed 30 occurrences, verified the fallback semantics, rewrote the plan, and resumed the same thread. Round two: VERDICT: APPROVED.

Did the factory perform as designed?

The answer is yes, for specific and inspectable reasons.

Evaluation verdict

Skill Factory v2 performed as designed.

On its first script-backed workflow skill, baseline-first evidence, frozen fixtures, and refusal gates demonstrably changed the outcome versus free-handing: the gate caught and removed a regression that the competent baseline shipped.

The caveats, without sanding them down

A successful stress test is not the same thing as a universal proof.

  • The builder also occupied the operator seats. Picking the guinea-pig task and judging the side-by-side are intended as builder-judges steps. This run was autonomous, so those judgments are recorded for an operator to overrule; they are not independent human validation.
  • The with-skill arm cost more at this size: about 281k orchestrator tokens versus about 117k for the baseline. The loop paid in quality and vendor diversity here, not token savings. Its economics improve only on larger, labor-heavy tasks; the skill’s applicability test is the guard against using it on trivial work.
  • The build machine did not have shellcheck. It was installed mid-run. The factory’s verification contract assumed the tool existed, so the environment had to be repaired before the promised check could pass.
  • The authoring process itself produced one bug. A malformed code fence in SKILL.md was caught and fixed before the side-by-side gate. It did not escape, but it is still evidence that the draft was not clean on first write.
  • The baseline was not bad. It passed the entire suite and produced a defensible implementation. The result supports the narrower claim that adversarial review improved this task’s outcome — not that every two-model run will.
